# Layout Todo

本文件是 layout 实施期的工作清单，不是正式规则文档。
它用于承接“已经定下方向，但不适合继续塞进规则文档”的实现排期与阶段性拆解。
当前不接入根 `README` 或 `docs/README`，避免它变成默认规则入口。

## 排序原则

layout 相关任务默认按以下维度综合排序：

### 1. 阻塞性

不做它，后续一串任务都无法稳定开展。
这类任务永远优先。

### 2. 本地可用性

即使完全没有 AI 能力，主业务也必须成立。
凡是决定主业务是否能独立运行、降级是否成立的任务，优先级高于大多数体验增强项。

### 3. 依赖完备度

上下游接口清楚、边界明确、技术成熟、做完不容易返工的任务，应优先进入主线。

### 4. 低风险顺手项

在不锁死后续方案的前提下，成本低、收益明确、顺手能完成的任务，可以排在主线之后快速落地。

### 5. 验证杠杆

做完后能够立刻验证大量后续工作、支撑 E2E 或暴露架构问题的任务，优先级应上调。

### 6. 探索后置

`Omni-bar`、`Artifacts Canvas`、视觉牵引线、复杂 motion、近场微干预等探索项，默认放在主链路稳定之后。

## 当前结论

当前阶段不再继续扩展规则文档，直接转入最小可运行 layout 的实现。

默认先做：

1. 不依赖 AI 也成立的主业务骨架
2. layout 基础交互与分层治理
3. 能支撑 E2E 的最小主链路

默认后做：

1. `Omni-bar`
2. `Artifacts Canvas`
3. 复杂动效与视觉牵引
4. 近场微干预

## 阶段风险提醒

以下提醒不是新的规则文档，而是当前实现路线里已经确认需要防范的工程风险。

### 1. P0 最容易低估的不是布局，而是键盘与焦点竞争

真正容易翻车的点不是 Flex 或 Grid，而是：

- Sidecar 打开时的焦点归属
- `Esc` 是先关 `main` 内部弹层，还是先关 Sidecar
- 组件库自带的焦点管理与 layout 层事件监听互相抢夺

因此在 P0 阶段，只要求预留统一注册接口，不要求一次完成完整的键盘快捷键栈。

真正的集中式快捷键栈放到 P1 跟进；P0 只允许非常克制的临时处理，且不得与 `main` 的原生弹层冲突。

P1 开始前先固定的优先级约定：

- `main` 内阻断型 Modal 优先于所有其他快捷键
- `main` 内局部浮层（如 Select、Popover、Dropdown）优先于 Sidecar
- Sidecar 高于页面级全局快捷键
- 页面级全局快捷键最低
- 后续统一通过注册栈管理，不允许各组件继续裸绑全局监听抢占优先级

### 2. P1 最危险的问题是状态来源失控

一旦 Sidecar、未来的 `Omni-bar`、`Artifacts Canvas` 各自维护会话状态，后续同步会非常痛苦。

因此需要尽早建立：

- `Session State`：会话、消息、生成中、错误
- `View State`：Sidecar 是否打开、当前视图形态、第三工作区是否展开

这两类状态必须分离，不能混写进同一个临时 UI store。

当前前置约定：

- 在真实 AI 能力接入前，availability 先以 mock 形式提供
- 当前默认使用 `unavailable` 作为降级源，先跑通 UI 壳层与主业务不受影响的链路
- 后续切换为真实状态源时，不改变 `Session State / View State` 的拆分方向

### 3. P1 以后不能再只靠 viewport media query

这套 layout 的关键变量不是浏览器宽度，而是 `main` 的真实可用宽度。

Sidecar 挤压后，视口没变，但 `main` 已经变窄。

因此后续业务区响应式调整，必须优先考虑：

- `@container`
- 或 `ResizeObserver` 配合容器宽度状态

### 4. P3 的视觉牵引需要更早预留 Overlay 能力

如果未来真要做跨区域连线、视觉牵引、跨层高亮，不能等到 P3 再临时补结构。

更稳的做法是在 P2 就预留全局 Overlay 挂载层或绘制层。

### 5. E2E 必须从一开始就考虑禁用动效

复杂动效一旦进入，E2E 很容易变 flaky。

因此在主链路 E2E 建立时，就应预留全局“禁用动效”开关，用于测试环境强制关闭过渡和动画。

## P0 阻塞项

### [x] 1. 抽出真正的 `app layout` 骨架

目标：

- 把当前路由壳层收敛为独立 layout 模块
- 明确 `main`、顶部轻量控制层、Sidecar 挂载位
- 确保 Sidecar 挂在 `<Outlet />` 之外

完成标准：

- `app/router` 不再堆叠过多壳层细节
- `main` 与 Sidecar 的职责边界在代码中清晰可见

### [x] 2. 先做“无 AI 也成立”的主链路

目标：

- 主业务页面在没有模型、没有问答、没有自动化时仍正常运行
- 协作入口可禁用、隐藏或说明，但不能破坏主导航

完成标准：

- 关闭或禁用 AI 后，首页、`labs`、`sandbox` 的基础路由与页面仍可用
- 不出现“必须依赖 Sidecar 才能完成基础浏览”的结构

### [x] 3. 建立 Z 轴语义化 token

目标：

- 把 `main`、`main` 内部弹层、Sidecar、Sidecar 内部浮层、治理层浮层的 `z-index` 分段切清

完成标准：

- layout 相关代码不再出现随手写的裸 `z-index`
- 后续浮层都通过统一 token 进入层级体系

### [x] 4. 完成 Sidecar 基础生命周期

目标：

- 打开/关闭
- 焦点转移与回退
- 路由切换不重置会话壳层
- 预留统一键盘注册接口

完成标准：

- Sidecar 可稳定打开和关闭
- 关闭后焦点回到触发入口
- 页面切换时 Sidecar 不被默认卸载
- P0 阶段允许使用克制的临时 `Esc` 处理，但不得与 `main` 的原生 Modal/弹层冲突
- 不在单个组件内长期保留裸绑 `window` 级快捷键监听
- 为后续集中式快捷键栈保留明确接入点

当前状态：

- 已满足 P0 完成线：打开/关闭、首焦点、关闭回焦、路由外持久挂载、基础键盘 E2E 均已建立
- 完整的集中式快捷键栈不在 P0 继续扩展，转入 P1 的独立任务跟进

### [x] 5. 为 Sidecar 共享状态预留最小外部宿主

目标：

- 在 P0 阶段就避免把跨组件状态锁死在 Sidecar 内部

完成标准：

- Sidecar 开闭状态不直接写死在组件内部 `useState`
- 至少通过最小 Context 或外部 store 承载跨组件共享状态
- 当前宿主只需覆盖最小状态，不要求在 P0 一次完成完整的 `Session State` / `View State` 拆分

## P1 主线增强项

### [x] 6. 分离 `Session State` 与 `View State`

目标：

- 在真正接入更多入口前，先把会话状态和界面状态拆开

完成标准：

- 会话消息、生成状态、错误状态不直接耦合到某一个 UI 入口
- Sidecar、未来的 `Omni-bar`、未来的第三工作区可共享同一会话层
- Sidecar 是否打开、宽度、第三工作区是否展开等仍保留为视图层状态

当前状态：

- `CollaborationSession` 已承接共享会话状态，不再绑定到 Sidecar 命名
- availability 已从 view state 移出，归入协作会话层
- 输入草稿已回收为本地状态，不再通过全局 session 高频驱动重渲染
- Sidecar 开闭与实测宽度仍保留在独立的 view state 中

### [x] 7. 建立集中式快捷键栈

目标：

- 把 `Esc`、全局协作入口快捷键等统一纳入集中式管理

完成标准：

- Sidecar 与 `main` 内部弹层不再靠临时事件碰撞决定优先级
- 快捷键注册和注销通过统一机制管理
- 后续 `Omni-bar` 或其他全局入口可复用同一套注册能力
- 优先级表与当前实现一致：
  - Modal > 局部浮层 > Sidecar > 页面级快捷键

当前状态：

- 已落最小集中式快捷键栈，统一注册与注销不再散落在单个组件里
- 当前已接入：
  - 页面级入口打开快捷键
  - Sidecar 关闭快捷键
- 后续 `Modal` / 局部浮层 / `Omni-bar` 可继续沿同一栈接入，不改变现有方向

### [x] 8. 顶部轻量控制层落为真实结构

目标：

- 顶部不只是占位 header
- 至少承载环境信息、少量一级入口、Sidecar 触发入口

完成标准：

- 顶部层不再只是临时文案拼接
- 后续可自然接入轻量协作入口

当前状态：

- 顶部已承载品牌、环境、角色与一级入口
- `开始` 入口当前保留为右下角全局触发位，顶部控制层负责提供稳定的上下文与导航骨架
- 已预留快捷键提示位，后续可自然承接 `Omni-bar` 或其他轻量入口

### [x] 9. 入口面板的默认态 / 增强态 / 只读态

目标：

- 明确入口面板的状态分层，而不是继续按“AI 是否可用”单独命名
- 让 Sidecar 同时容纳：
  - 默认欢迎与语义输入
  - 本地入口模式
  - 增强能力模式
  - 仅保留历史的只读模式

完成标准：

- 各状态的 UI 区分清楚
- 不需要真实 AI 也能把入口面板跑通
- 当前前置状态已落为 mock unavailable；后续应将其收敛为“增强能力未连接，但本地入口仍可用”的语义
- 至少补齐：
  - 默认欢迎态
  - 本地入口态
  - 增强能力可用 / 降级态
  - 历史只读态

当前状态：

- 入口面板已支持 `available / degraded / readonly / unavailable` 四类 mock availability
- `unavailable` 与 `degraded` 已在 UI 上明确区分，且都能继续走本地入口链路
- `readonly` 已落为只读浏览态，并禁用新的输入
- 对应 smoke E2E 已覆盖 degraded 与 readonly

### [x] 10. 建立 `main` 容器真实宽度响应机制

目标：

- 不再只依赖 viewport 宽度判断业务区的响应式
- 让 `main` 在被 Sidecar 挤压时仍能正确触发内部布局切换

完成标准：

- 对关键业务容器，优先使用 `@container` 或等效容器宽度观测机制
- Sidecar 打开后，`main` 内部关键内容不会因为“视口没变”而错过响应式切换
- 同时提供基于 `ResizeObserver` 的 fallback，用于向样式层注入真实宽度变量或等效状态
- 不把主链路完全绑定在纯 CSS 容器查询上

当前状态：

- `main` 与 `sidecar` 都已通过 `ResizeObserver` 暴露真实宽度变量与 band
- Sidecar 打开时，layout 会根据实测 sidecar 宽度为 `main` 与顶部控制层真实让位
- Sidecar 已去除遮罩并允许主区继续交互，主链路 E2E 已覆盖“sidecar 打开时主区仍可跳转”

### [x] 11. 补主链路 E2E

目标：

- 用 E2E 固定 layout 最低可用线

完成标准：

- 至少覆盖：
  - `main` 在无 AI 下可用
  - Sidecar 可打开/关闭
  - AI 不可用时不破坏主业务
  - 路由切换不直接摧毁 Sidecar 壳层
- 测试环境存在统一的禁用动效开关，避免动画导致的 flaky
- 优先通过在 `html` 或 `body` 上注入统一类名，并使用全局样式强制关闭 `animation` / `transition`
- 不依赖散落在各业务组件中的 `if (isTest)` 逻辑去逐个关闭动画

当前状态：

- 已覆盖：
  - `main` 在无增强能力下可用
  - Sidecar 可打开/关闭
  - 增强入口不可用时不破坏主业务
  - 路由切换不直接摧毁 Sidecar 壳层，且切换后仍可继续使用本地入口
- E2E 已通过统一 `e2e/test.ts` 在 `html` 上注入 `disable-motion` 类
- 全局样式已统一关闭测试环境下的 `animation` / `transition`
