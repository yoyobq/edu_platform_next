已完成的 P0 / P1 / P2 主线项已从本文件移出，避免待办清单继续混杂历史完成记录。

## 当前结论

当前这轮 layout 验证可以视为阶段性完成。

已确认的前提：

- 当前首页仍是 demo / workbench 形态，不是最终登录后入口页
- Sidecar、Overlay、第三工作区、Omni 触发位的最小结构都已验证过
- `labs/demo` 与主页触发词已经足够验证“跳层”这件事是否成立

因此下一阶段不再继续扩展当前 demo 首页上的 layout 复杂度，而是优先等待真实入口页与登录链路出现后再决定下一步。

## 已确认风险

以下提醒仍然有效，后续进入下一轮实现时应继续遵守：

### 1. 键盘与焦点竞争仍是高风险区

- `main` 内阻断型 Modal 优先于所有其他快捷键
- `main` 内局部浮层优先于 Sidecar
- Sidecar 高于页面级全局快捷键
- 页面级全局快捷键最低

### 2. `Session State` 与 `View State` 不能重新混写

- 会话、消息、错误、生成状态继续留在会话层
- Sidecar、第三工作区、未来 Omni 的开关与形态继续留在视图层

### 3. 业务响应式仍应优先看真实可用宽度

- Sidecar / 第三工作区对 `main` 的挤压不能退回成只看 viewport
- 优先继续使用容器宽度或等效观测方式

### 4. E2E 继续默认禁用动效

- 后续只要引入新的 layout 动效，测试环境仍必须可统一关闭

## 决策前置项

这些事项优先级高于继续做 P3 探索，因为它们会决定后续 layout 是否需要换壳或收缩范围。

### [ ] 1. 真实入口页出现后，是否拆成两套 layout 壳层

目标：

- 明确“登录 / 真正入口页”和“登录后工作台”是否继续共用当前 `AppLayout`

为什么优先：

- 这会直接影响 Sidecar、第三工作区、Omni host 是否还应该常驻首屏
- 如果未来入口页更轻，当前这套工作台壳层不应强行前置到登录前

完成标准：

- 明确是单壳继续演进，还是拆成“入口壳 + 工作台壳”两套
- 明确当前全局 host 哪些继续常驻，哪些只在工作台壳层挂载

### [ ] 2. 第三工作区的正式状态来源是否继续使用 URL

目标：

- 决定当前 demo 用的 URL search 方案是否继续保留，还是迁回独立 `View State`

为什么优先：

- 现在 URL 方案很适合 demo 验证与跨页面触发
- 但正式进入多工作区形态后，是否还适合承载打开态、当前 artifact、退出路径，仍未定

完成标准：

- 明确 demo 态和正式态是否共用状态来源
- 明确第三工作区后续是否进入独立 view state

### [ ] 3. Omni 入口的正式角色定义

目标：

- 明确 Omni 入口是“轻命令入口”，还是“全局搜索 + 命令 + 最近操作”的复合入口

为什么优先：

- 这会影响顶部结构、快捷键栈、输入语义和后续实现复杂度

完成标准：

- 明确 Omni 的职责边界
- 明确它与 Sidecar 的关系是补充还是分流
