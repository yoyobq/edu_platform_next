# Workbench Entry Plan

本计划用于明确“正式首页 / 工作台入口 / 登录后默认结构”的方向。

它不是具体页面稿，也不是登录实现清单，而是下一阶段的产品与架构边界说明。

## 目标

- 明确首页不是传统欢迎页，而是登录后用户的默认工作台
- 明确 `layout`、首页工作台、正式功能页三者的边界
- 为后续登录体系、用户态和真实业务入口提供稳定方向

## 当前判断

- 当前 `AppLayout` 已经承担全局壳层职责
- 开始按钮、顶栏、Sidecar 继续视为全局工作台骨架
- 当前首页仍是过渡态，不代表正式登录后入口
- 当前 API 状态面板可作为暂时的首页内容，但不应被误判为最终首页信息架构

## 核心方向

### 1. 首页是工作台，不是欢迎页

- 登录后的 `/` 默认进入用户工作台
- 工作台用于承接高频入口、关注能力、最近上下文和少量状态概览
- 首页不追求替代所有页面，也不要求每个页面都缩成一个“小窗”
- `/` 作为登录后默认工作台入口；未登录用户访问时，由路由守卫或入口层决定跳转登录页，或进入独立的未登录展示页
- 未登录展示页不属于工作台信息架构，也不默认复用完整工作台壳层

### 2. 全局壳层与首页内容分开

- 顶栏、开始按钮、Sidecar、未来 Omni 继续留在 `layout`
- 首页工作台内容继续留在 `pages/home`
- 只有确实具备跨页面持续价值的状态或入口，才上升为全局壳层的一部分

### 3. 首页模块是摘要和入口，不是页面缩略图

- 首页上的模块更像“工作单元摘要”
- 它们可以展示状态、最近项、快捷动作、进入正式页面的入口
- 不要求每个正式页面都先有一个窗口级组件或桌面缩略版

### 4. 用户态决定首页编排

- 首页默认出现哪些模块，应由登录后的用户态决定
- 这里的用户态可以影响默认首页编排，但不等于把权限、默认编排和个人偏好混成同一套机制
- 后续可先从“常用功能 / 关注模块”这类轻量偏好开始，不必一开始就做完整个性化系统

补充边界：

- 权限：决定某项能力是否可见、是否可进入
- 默认编排：基于身份、角色或产品策略决定登录后的初始首页结构
- 个人偏好：由用户自己调整顺序、关注模块或常用入口
- 这三者相关，但不能在接口和实现上混成同一层语义
- 若无用户自定义偏好，首页应先降级为基于角色的默认工作台模板
- 若角色模板也暂未建立，仍应提供一个最小默认工作台，避免首页退化为空白页

### 5. 首页模块必须有准入理由

- 首页模块不是默认展示位，不应因为“某个页面存在”就自动进入首页
- 至少满足以下条件之一，才考虑进入首页：高频使用、需要持续关注、能提供明确下一步动作、能承接最近上下文、能明显减少一次跳转成本
- 不满足这些条件的能力，优先留在正式页面，不进入首页工作台

## 结构边界

### layout

- 全局导航骨架
- 开始按钮
- Sidecar
- 全局协作入口
- 未来必要的全局状态位

### pages/home

- 用户默认工作台
- 工作台模块编排
- 首页只承载轻量概览与快捷入口，不承载完整流程
- 首页模块应能在有限空间内完成“识别状态 + 触发动作”
- 复杂筛选、长列表、深度编辑等完整能力继续进入正式页面
- 工作台上的每个模块默认只作为 UI Adapter，不直接承载复杂业务逻辑
- 模块需要的数据和摘要，应优先通过对应 feature 暴露的 use case 获取
- 若某个首页模块开始承载复杂查询、流程编排或业务判断，应下沉为独立 feature，再由首页消费其公开入口

### 其他正式页面

- 承接完整业务能力
- 不因为首页存在而被迫提供缩略版或窗口版

## Sidecar 与页面联动边界

- Sidecar 继续属于全局壳层，不直接改写页面私有状态
- 页面可向全局协作层暴露受控的 context / state snapshot，供 Sidecar 读取、总结和辅助导航
- 若 Sidecar 需要影响某个页面的表单、筛选、局部流程或其他可写状态，必须通过该页面预先声明的受控 action 进行
- 这套联动默认从“可读上下文”开始，不默认要求每个页面一开始就提供完整动作集合
- 只有确实需要联动的页面，再逐步补充受控 actions 和更深层的协作能力
- 不把“页面协作接口”误做成一开始就覆盖所有页面的重体系

## 前置条件

### 1. 先有登录态，后有真正的工作台

- 没有登录态，就无法稳定判断用户是谁、能看什么、默认展示什么
- 因此首页工作台的正式落地，优先级依赖最小登录链路与用户信息来源

### 2. 先明确默认入口，再扩展页面级业务接入

- 在默认入口还不稳定时，不继续扩展首页为杂糅式 demo workbench
- 后续真实业务接入，应优先围绕“登录后默认工作台 + 正式页面入口”展开

## 当前不做的事

- 不把首页做成传统门户或纯欢迎页
- 不把每个页面强行改造成桌面窗口
- 不把所有状态都提前提升到 `layout`
- 不在登录体系未出现前，过早拍死首页模块清单

## 下一阶段顺序

1. 明确最小登录链路与用户信息来源
2. 明确登录后默认入口仍为 `/`
3. 为首页工作台定义第一批模块类型：状态概览、常用入口、最近上下文、快捷动作
4. 再决定哪些模块只属于首页，哪些值得上升为全局壳层能力

## 完成标准

- 团队对“首页、全局壳层、正式页面”的边界有统一说法
- 后续登录体系和首页改造不再互相牵扯
- 新增页面时，不再默认要求先做桌面缩略态
