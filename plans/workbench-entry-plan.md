# Workbench Entry Plan

本计划用于推进“正式首页 / 工作台入口 / 登录后默认结构”的落地顺序。

其中已经稳定的主题边界，已迁入 [docs/workbench-entry-rules.md](../docs/workbench-entry-rules.md)。

本文件不再承担正式规则说明，只保留当前要推进的事项、前置条件和阶段顺序。

## 范围说明

- 本次拆分只适用于 `workbench entry` 主题
- 它不构成仓库内所有模块统一适用的 plan 分层方法

## 目标

- 让 workbench entry 主题从“方向说明”进入“可执行推进”
- 把已经稳定的边界从 plan 中移出，避免后续实现继续把 plan 当规则来源
- 明确哪些事必须先做，哪些事可以先做，哪些事应等待前置条件成熟

## 已移入规则的内容

以下内容已进入正式主题规则，不再在本 plan 中重复展开：

- 登录后的 `/` 默认进入工作台，而不是欢迎页
- 未登录展示页不属于工作台信息架构
- 全局壳层与首页工作台内容分开
- 首页模块是摘要与入口，不是正式页面缩略图
- 权限、默认编排、个人偏好不能混成同一语义层
- 首页模块必须有准入理由
- 首页模块默认只作为 UI Adapter
- Sidecar 与页面联动必须保持受控边界

## 当前现状

- 当前 `AppLayout` 已经承担全局壳层职责
- 开始按钮、顶栏、Sidecar 仍在验证全局工作台骨架
- 当前首页仍是过渡态，不代表正式登录后入口
- 当前 API 状态面板可作为暂时首页内容，但不应被误判为最终首页信息架构

## 当前前置条件

### 1. 先有最小登录链路，后有真正工作台

- 没有登录态，就无法稳定判断用户是谁、能看什么、默认展示什么
- 因此首页工作台的正式落地，依赖最小登录链路与用户信息来源

### 2. 依托确定的入口再扩展承载面

- 既然 `/` 已经明确为登录后默认入口，就不应再继续把当前首页扩展为杂糅式 demo workbench
- 后续真实业务接入，必须优先围绕“登录后默认工作台 + 正式页面入口”展开

## 优先级定义

- `P0`：不做就会阻塞 workbench entry 主线落地
- `P1`：不阻塞主线，但应尽早补齐，否则实现容易继续发散
- `P2`：需要首页与正式页面开始稳定出现后，再做判断
- `P3`：需要更多真实使用模式或更成熟前置条件后，再展开

## P0

### 1. 明确未登录入口的独立壳层方案

- 规则已明确：未登录展示页不默认复用完整工作台壳层
- 本阶段需要明确“未登录展示页 / 登录页”具体采用什么独立壳层方案
- 明确哪些全局 host 继续常驻，哪些只属于登录后工作台壳层
- 这是后续登录接入和 `/` 路由治理的前置决策
- 产出物：一份未登录入口壳层方案说明，至少回答路由挂载位置、是否复用现有 `AppLayout` 的局部能力、以及各类 host 的归属边界
- 当前产出见 [workbench-entry-unauth-shell.md](./workbench-entry-unauth-shell.md)

### 2. 明确最小登录链路与用户信息来源

- 先明确当前项目里最小可落地的登录态
- 先明确用户身份、角色、能力范围由谁提供
- 在这一步完成前，不把 URL 上的临时 query 语义继续外溢成正式产品契约
- 产出物：一份最小登录链路说明，至少回答身份来源、权限来源、默认编排来源分别由谁提供，以及它们在前端的接入边界
- 当前产出见 [workbench-entry-min-auth.md](./workbench-entry-min-auth.md)

### 3. 围绕已确定的 `/` 默认入口推进实现

- 登录后的 `/` 默认进入工作台，已是正式规则，不在本 plan 中重新开放
- 后续实现应围绕这个默认入口补齐登录接入、用户态来源和首页工作台承载面
- 若未来真的需要改动默认入口，应先修改 rule，再回写 plan，而不是在 plan 内直接保留反向分支
- `P0-3` 的稳定实现依赖 `P0-2` 先提供可恢复的 session 判断基础，例如 session restore、已登录态判断与会话失效回退
- 因此 `P0-3` 可以先设计分流语义，但真正落地 protected / public 分支守卫前，应先完成 `P0-2` 的最小认证基础
- 产出物：一份默认入口接入清单，至少覆盖登录后跳转路径、未登录拦截或展示路径、以及 `/` 在前后态下的职责说明
- 当前产出见 [workbench-entry-root-routing.md](./workbench-entry-root-routing.md)

## P1

### 1. 为首页工作台定义最小模块 contract

- 明确首页模块最小需要回答什么：摘要数据、主动作、空态、错误态、不可见策略、进入正式页面的跳转位
- 该 contract 只服务 workbench 首页，不扩展成仓库级通用模块规范
- 产出物：一份首页模块最小 contract，至少包含摘要数据、主动作、空态、错误态、可见性规则、跳转位这几类统一结构
- 当前产出见 [workbench-entry-home-module-contract.md](./workbench-entry-home-module-contract.md)

### 2. 标记并隔离当前 demo 语义

- 把当前首页、Sidecar、第三工作区中的 demo 触发逻辑明确标成过渡验证用途
- 避免它们继续被误当成正式工作台语义向外扩散
- 至少从以下维度明确隔离方式：命名隔离、路由隔离、文案标识隔离、目录归属隔离
- 命名上避免继续使用容易被误解为正式能力的名称
- 路由上避免把 demo 验证入口继续混进正式默认入口语义
- 文案上明确哪些能力是 demo、过渡验证或实验能力
- 目录归属上避免 demo 逻辑继续长进正式首页工作台语义层
- 产出物：一份 demo 语义清单，标明哪些逻辑属于过渡验证、分别通过什么方式隔离、后续准备迁移到哪里、哪些不进入正式工作台语义
- 当前产出见 [workbench-entry-demo-semantics-isolation.md](./workbench-entry-demo-semantics-isolation.md)

### 3. 准备最小默认工作台兜底

- 在没有角色模板、没有个人偏好时，先有一个最小默认工作台
- 该兜底只承接最基础的状态概览、常用入口或最近上下文
- 产出物：一份最小默认工作台草案，至少回答默认出现哪些模块类型、空白用户首次进入时看到什么、以及缺少偏好配置时如何降级
- 当前产出见 [workbench-entry-default-workbench-fallback.md](./workbench-entry-default-workbench-fallback.md)

## P2

### 1. 第一批正式首页模块清单

- 等最小登录链路、默认入口和更多正式页面出现后，再确认第一批正式模块
- 避免当前只根据过渡态首页或 demo 页面反推正式模块清单
- 产出物：一份首批首页模块名单，并标注每个模块的准入理由、数据来源和对应正式页面入口

### 2. 哪些模块上升为全局壳层能力

- 等首页模块与正式页面都出现稳定使用模式后，再判断哪些入口或状态值得上升到 `layout`
- 不提前把首页试验项提升成全局常驻能力
- 产出物：一份壳层提升清单，标明哪些能力留在首页，哪些能力上升到 `layout`，以及上升理由

## P3

### 3. 更深层的 Sidecar 页面联动

- 先从只读上下文和受控导航开始
- 更深的页面 action、表单联动、局部流程协作，等待真实页面提出明确需求后再补
- 产出物：一份页面协作接口清单，按页面列出已开放的只读上下文、受控 action 和明确不支持的联动范围

### 4. 默认编排与个人偏好的细化机制

- 当前先保证“权限 / 默认编排 / 个人偏好”三者分语义
- 具体偏好存储、排序编辑、关注模块管理，等待更多首页模块和真实使用模式后再定
- 产出物：一份编排机制说明，明确默认编排与个人偏好的数据边界、覆盖关系和存储位置

## 当前不做的事

- 不把首页做成传统门户或纯欢迎页
- 不把每个页面强行改造成桌面窗口
- 不把所有状态都提前提升到 `layout`
- 不在登录体系未出现前，过早拍死首页模块清单

## 当前推荐顺序

1. 明确未登录入口的独立壳层方案
2. 明确最小登录链路与用户信息来源
3. 围绕已确定的 `/` 默认入口推进实现
4. 为首页工作台定义最小模块 contract 与最小默认模板
5. 再确定第一批正式首页模块类型
6. 最后决定哪些模块只属于首页，哪些值得上升为全局壳层能力

## 完成标准

- plan 不再产出稳定产品规则；稳定规则只进入 `docs/` 下对应 rule 文档
- 已稳定的边界不再通过 plan 口口相传，而有正式 rule 可依
- 团队对“首页工作台、全局壳层、正式页面”的边界有统一说法
- 后续登录体系和首页改造不再互相牵扯
- workbench entry 可以按阶段推进，而不再继续扩展成杂糅式 demo 首页
