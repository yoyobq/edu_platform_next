# Workbench Entry Minimal Default Workbench Proposal

本文件是 `workbench entry` 主题下 P1-3 的当前产出物。

它只回答以下问题：

- 在没有角色模板、没有个人偏好时，登录后首页至少应怎样兜底
- 最小默认工作台应优先承接哪些模块类型
- 空白用户第一次进入时，应看到什么而不是空白页

它不回答：

- 第一批正式首页模块名单
- 个人偏好的具体存储结构
- 默认编排编辑器与拖拽排序交互

这些内容继续分别属于 P2、P3 与后续实现阶段。

## 目标

- 让登录后的 `/` 在角色模板尚未建立前，仍有一个稳定、克制、可用的首页兜底
- 避免首页因为缺少真实模板或偏好配置，继续被 demo 内容临时填满
- 让 P1-1 的单模块 contract 与 P0 的登录后默认入口语义，在页面级首次形成可执行连接

## 当前判断

- P1-1 已经定义单模块最小 contract，但还没有回答“没有模板时首页整体如何落地”
- 当前首页仍是过渡态 API 状态面板，不能直接当作正式默认工作台
- P0-2 已明确：首阶段默认编排来源应是前端维护的角色模板映射；但在角色模板还没建立前，仍需要最小兜底
- P1-2 已明确：demo trigger、demo bridge、`workspaceDemo` 等语义不应用来填补正式首页空位
- 因此 `minimal-default` 不只是“都还没有时的临时占位”，而是早期可直接承担首阶段默认模板、后续也继续保留的终端 fallback

## 结论

最小默认工作台不是“当前首页先放什么算什么”，而是一份页面级兜底模板。

它的定位应同时满足两层：

- 在角色模板尚未建立前，可临时直接承担首阶段默认模板
- 在角色模板已经建立后，继续作为最终 fallback 保留，而不是被删除

它至少包含三类模块类型：

1. 状态概览
2. 常用入口
3. 最近上下文

一句话说：

- 先让用户知道“现在系统状态如何”
- 再让用户知道“下一步最常见能做什么”
- 最后在有真实数据时，再补“最近上下文”

## 为什么是这三类

### 1. 状态概览

作用：

- 承接少量系统或账号相关的轻量摘要
- 给首页一个稳定的“抬头信息区”，避免首次进入时完全没有可识别内容

边界：

- 只做概览，不做长列表
- 只做摘要与主动作，不承接完整排障流程

当前最接近的过渡参考：

- `ApiHealthStatusPanel`

但当前结论仍是：

- 它可以提供“状态概览类模块”的参考方向
- 不直接等于正式默认工作台必须长成 API 面板

### 2. 常用入口

作用：

- 承接首次进入时最容易理解的下一步动作
- 保证用户在没有个性化偏好时，也能快速进入 1 到 3 个正式页面或稳定入口

边界：

- 只放正式页面入口或稳定受控动作
- 不把 `labs / sandbox` 试验页当成默认正式入口
- 不把 demo bridge、触发词、`workspaceDemo` 这类过渡语义塞进默认工作台
- 该槽位应保证至少存在一个“始终安全、始终可见、当前环境可达”的 CTA，避免纯白板用户首屏没有任何可点击的下一步

### 3. 最近上下文

作用：

- 承接“最近看过什么 / 正在跟进什么 / 最近一次停在哪里”这一类真实上下文
- 让首页不仅是静态入口页，也能在有数据时变成登录后的回到点

边界：

- 没有真实上下文时，允许以空态出现
- 不拿 Sidecar demo 对话、第三工作区 demo artifact 或 URL 模拟参数冒充最近上下文

## 当前推荐的最小默认模板

页面级最小默认模板当前建议如下：

```ts
type MinimalDefaultWorkbenchTemplate = {
  key: 'minimal-default';
  slots: readonly ['status-overview-slot', 'primary-entry-shortcuts-slot', 'recent-context-slot'];
};
```

说明：

- `status-overview-slot` 必须存在
- `primary-entry-shortcuts-slot` 必须存在
- `recent-context-slot` 默认保留，但允许在可见时以空态出现
- 这里定义的是页面级 fallback 槽位类型，不等于 P2 已经确定的正式模块 ID 或首批模块清单

这意味着最小默认工作台的目标不是“凑够很多卡片”，而是先保证：

- 有状态锚点
- 有明确下一步
- 有面向回访用户的增长位

## 空白用户首次进入时应看到什么

这里的“空白用户”指：

- 已完成登录
- 还没有个人偏好
- 也还没有可用的最近上下文
- 当前角色模板暂未建立或暂不可用

这时首页不应为空，也不应继续落回 demo 首页。

当前建议首屏看到：

1. 一个状态概览模块
2. 一个常用入口模块
3. 一个最近上下文空态模块

用户感知上应回答三件事：

- 你已进入登录后工作台
- 你现在可以先从这些稳定入口继续
- 你目前还没有最近上下文，后续使用后这里会逐步出现内容

不建议的做法：

- 只给一个空白容器
- 继续拿 API 面板充当全部首页
- 用 demo bridge 或实验页入口填补正式首页空位

## 缺少偏好配置时的降级逻辑

当前推荐把首页初始化收束成以下顺序：

```txt
已登录 session
  -> 读取个人偏好
  -> 若有可用偏好，使用个人偏好
  -> 若无个人偏好，读取角色模板
  -> 若有角色模板，使用角色模板
  -> 若无角色模板，回退到 minimal-default template
```

其中：

- 个人偏好回答“这个用户后来怎么改”
- 角色模板回答“这类用户首次进入时优先看什么”
- 最小默认模板回答“都还没有时，至少别让首页失语”
- 在角色模板尚未建立的阶段，`minimal-default` 也可以直接承担首阶段默认模板职责
- 在角色模板建立之后，`minimal-default` 仍继续作为终端 fallback 保留，slot 结构不因角色模板上线而撤销

这三层依然不能混写。

## 模块级降级要求

即使进入最小默认模板，每个模块仍继续服从 P1-1 contract。

当前建议：

- 状态概览模块：无数据时进入空态或错误态，但模块本身仍可见
- 常用入口模块：若某些正式入口当前不可见，应按可见性规则过滤，只展示当前用户可进入的稳定入口；同时应保留至少一个“绝对安全且始终可见”的兜底 CTA
- 最近上下文模块：若该能力对当前用户可见但暂无数据，优先显示空态；若该能力当前阶段未接入或对当前用户不可见，仍允许不渲染

原因：

- 在“可见但暂无数据”的情况下，保留空态更能表达“这里未来会承接真实上下文”
- 但是否渲染该模块，仍应继续服从 P1-1 中 `visibility` 与 `state` 分层表达
- 常用入口区若被过滤到完全为空，会让首屏看起来像异常或未完成，因此需要至少一个稳定兜底动作保持 CTA 闭环

## 当前不应进入最小默认模板的内容

- `labs` 页面默认入口
- `sandbox` 页面默认入口
- demo trigger
- `workspaceDemo` query 相关能力
- 第三工作区 demo artifact
- Sidecar 中任何只服务验证链路的桥接动作

这些内容即使短期可用，也不应被当作“没有模板时先凑一下”的默认工作台组成。

## 对页面拥有者的直接要求

### `pages/home`

- 只消费已整理好的默认模板结果
- 不直接判断“该回退到哪个模板”
- 不自行把 demo 内容拼成首页模块

### 首页拥有者 / `features/workbench-entry`

- 负责给出：
  - 当前用户是否有个人偏好
  - 当前角色是否已有默认模板
  - 若两者都没有，应回退到哪个最小默认模板

### 认证切片

- 只提供已登录用户态、角色与权限来源
- 不直接决定首页模块排列

## 当前推荐实现顺序

1. 先建立页面级 `minimal-default` 模板定义
2. 让 `pages/home` 消费模板产出的模块列表，而不是继续手写单个过渡组件
3. 先用“状态概览 + 常用入口 + 最近上下文空态”组成首个兜底版本
4. 后续角色模板建立后，再把该最小模板降为统一 fallback，而不是删除

## 完成标准

- 已明确没有角色模板、没有个人偏好时的最小默认工作台应包含哪些模块类型
- 已明确空白用户首次进入时首页至少应看到什么
- 已明确个人偏好、角色模板、最小默认模板三层降级顺序
- 已明确 `minimal-default` 在角色模板建立前可临时直接承担默认模板、在角色模板建立后继续作为终端 fallback 保留
- 已明确 demo 语义不进入默认工作台兜底
