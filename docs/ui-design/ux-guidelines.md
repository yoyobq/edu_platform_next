# UX Guidelines

本文件包含两类内容，性质不同：

**正式规则**（应默认遵守）：

- 确认边界：高风险、破坏性、不可逆操作必须有显式确认
- AI 状态模型：AI 交互必须拆成 Thinking / Streaming / Complete 三态，不与普通 HTTP loading 混用
- 公共入口布局模式：登录前独立页面统一使用左右两栏结构（左说明 + 右操作）

**交互建议**（判断框架，非硬规则）：

- 交互容器选择（页面、Drawer、Modal、Popover、Tooltip 等场景判断）
- 反馈方式选择（message、notification、Alert 等）

整体目标是帮助人和 AI 在常见页面设计里快速做出更稳妥的选择。最终仍应服从真实需求、业务风险、用户频率、设备约束和流程目标。

若需求与交互建议冲突，以需求为准；若需求与正式规则冲突，应记录原因并经过讨论。

补充审美判断见 [brand/brand-tone.md](./brand/brand-tone.md)：同一视图只保留一个主声部，不通过额外装饰反复强调同一件事。

## 使用方式

适合用于：

- 生成页面或组件草案前做快速判断
- 在多个交互容器之间做一轮初筛
- Code Review 时判断当前交互是否明显过重或过轻

不适合用于：

- 把容器选择建议写成必须遵守的硬规则（容器选择属于判断框架，非强制规则）
- 代替业务流程分析
- 绕过 [layout.md](../layout.md)、[ui-stack-rules.md](../ui-stack-rules.md) 与正式需求

## 先看哪几个维度

决定交互形态前，先回答这几个问题：

### 1. 这是主任务，还是局部任务

- 主任务：更接近页面、独立路由、完整工作区
- 局部任务：更接近卡片内区块、局部表单、局部反馈

### 2. 用户是否需要保留当前上下文

- 需要持续看着原页面做判断：优先 `Drawer`、inline 区块、Sidecar
- 可以短暂打断当前上下文：才考虑 `Modal`

### 3. 这是查看、编辑，还是确认

- 查看：优先就地展示、展开区块、详情页、Drawer
- 编辑：看复杂度决定 inline / Drawer / 页面
- 确认：优先轻量确认，不要把确认做成完整页面

### 4. 操作是否可逆，风险是否高

- 高风险、破坏性、不可逆：必须显式确认
- 低风险、可撤销：优先轻反馈，减少阻断

### 5. 任务有多长、多复杂

- 3 秒能完成：优先轻容器
- 需要连续思考、输入、校对：不要塞进小弹层

### 6. 它是否值得被再次访问

- 需要分享、收藏、回访、刷新保留：更适合页面或有状态工作区
- 用完即走：更适合局部容器

## 项目已落地模式

以下是当前项目中已经实现的交互模式，新页面应优先参考这些已验证的模式。

### 公共入口页面：左右两栏

登录、找回密码、验证入口（`src/pages/login/`、`src/pages/forgot-password/`、`src/pages/verification-intent/`）统一使用左右两栏布局：

- 左栏：页面标题 + 上下文说明，帮助用户理解当前步骤的意义
- 右栏：`Card` 包裹的操作区，承载表单或状态反馈

约束：

- 右栏 Card 内的标题用 `Typography.Title level={4}`，不与左栏页面标题竞争
- 状态切换（如"发送邮件 → 检查邮箱"）在右栏 Card 内部完成，不切换路由
- 返回登录等导航动作用 `Button type="link"` 放在 Card 底部

### 全局协作入口：浮动按钮 + Drawer Sidecar

项目的 AI 协作入口采用浮动按钮（`✨ 开始`）触发右侧 Drawer（`src/app/layout/entry-sidecar.tsx`）：

- 浮动按钮固定在右下角，包含快捷键提示（`Alt+K`）
- Sidecar 基于 antd `<Drawer>` 实现，`mask={false}` 允许用户继续操作主区域
- 打开时 `main` 弹性收窄（桌面端），关闭时 `main` 回到全宽
- `Esc` 优先关闭 Sidecar，焦点回退到触发按钮

约束：

- Sidecar 关闭后，主业务不能失效
- Sidecar 内的导航卡片可将用户引导到目标页面，但不替代页面本身的主入口
- Drawer `title` prop 允许轻量 JSX（如图标 + 文案），但不嵌套 `Typography.Title`

### 工作台首页：模块化卡片

`src/pages/home/index.tsx` 采用模块化卡片（HomeModuleCard）展示工作台入口：

- 每张卡片独立承载一个功能模块的状态摘要与操作入口
- 卡片支持三种状态：ready（摘要数据）、empty（引导说明）、error（错误反馈 + 重试）
- 加载中使用 `<Skeleton>` 占位，不使用 `<Spin>` 覆盖

### 第三工作区跳层

当 AI 生成结果物超出 Sidecar 的舒适阅读范围时，通过 URL search 参数触发第三工作区面板（`src/shared/third-workspace-demo/canvas.tsx`）：

- 从 Sidecar 或 labs 页面触发，不是独立路由
- 打开后覆盖在 `main` 和 Sidecar 之上，用户可随时关闭回到原位
- 关闭按钮明确可见，不依赖 `Esc`（`Esc` 已被 Sidecar 使用）

## 正向建议

### 什么时候优先用页面

页面适合：

- 独立业务任务
- 需要回访、分享、收藏、刷新保留的内容
- 多步骤流程
- 长表单、复杂编辑、长阅读
- 主信息密度高的详情和工作区

页面不适合：

- 只做一次轻确认
- 只改一个小字段
- 只看一小段补充说明

### 什么时候优先用 inline 区块

inline 区块适合：

- 与当前内容强相关的补充信息
- 局部过滤、局部说明、局部编辑
- 单字段、低校验成本的编辑，如改名称、切状态
- 展开后仍希望用户继续留在当前阅读流里

inline 区块不适合：

- 多字段、有关联校验的编辑
- 需要明显隔离风险的操作
- 会把原页面撑得过长、过乱的复杂流程

### 什么时候优先用 `Modal`

`Modal` 适合：

- 简短确认
- 单步决策
- 短表单
- 需要暂时阻断背景交互的高注意力任务

典型场景：

- 删除确认
- 切换状态前确认
- 创建一个简单对象
- 解释某个关键风险并让用户确认继续

`Modal` 不适合：

- 长表单
- 需要频繁对照背景内容的编辑任务
- 需要分页、复杂筛选、长列表浏览
- 作为主详情页替代品

### 什么时候优先用 `Drawer`

`Drawer` 适合：

- 需要保留当前页面上下文的查看或编辑
- 中等复杂度表单
- 详情预览、审阅、补充操作
- 需要边看主页面边操作的任务

典型场景：

- 从列表查看详情
- 在不离开当前页面的前提下编辑一条记录
- 审阅 AI 建议、引用、补充信息

`Drawer` 不适合：

- 只是一次确认
- 任务已经复杂到接近完整页面
- 小屏下内容会被压缩到不可用

项目注意：当前 EntrySidecar 已占用右侧 Drawer 位置。若主业务也需要右侧 Drawer，应优先让 Sidecar 收窄或暂时关闭，避免两个同等级面板竞争注意力（参考 [layout.md](../layout.md) 动态行为原则）。

### 什么时候优先用 `Popover`

`Popover` 适合：

- 一小组补充动作
- 轻量解释
- 极短设置项
- 不值得开 `Modal` / `Drawer` 的局部操作

`Popover` 不适合：

- 长内容
- 长表单
- 关键确认
- 依赖稳定焦点管理的复杂交互

### 什么时候优先用 `Popconfirm`

`Popconfirm` 适合：

- 单条、行内、后果直观的轻量确认
- 表格单行删除、单行状态切换
- 用户已有足够上下文、不需要额外风险说明

`Popconfirm` 不适合：

- 影响多条数据的批量操作
- 需要向用户说明后果或风险的场景
- 高风险、不可逆操作（用 `Modal` 强制阅读提示）

### 什么时候优先用 `Tooltip`

`Tooltip` 适合：

- 补一句短解释
- 解释图标含义
- 提示禁用原因

`Tooltip` 不适合：

- 承载主信息
- 放操作入口
- 放多段文案

已落地参考：`src/app/layout/app-layout.tsx` 中 Omni-bar 触发按钮使用 `<Tooltip title="全局搜索与命令 (预留)">`。

### 什么时候优先用 `Dropdown`

`Dropdown` 适合：

- 同类动作集合
- 溢出操作
- 单选型轻菜单

`Dropdown` 不适合：

- 承载复杂表单
- 承载需要阅读理解的大段说明

### 什么时候优先用 `Tabs`

`Tabs` 适合：

- 同一对象下的并列视角
- 用户明确知道自己在切换"视图"而不是切换流程

`Tabs` 不适合：

- 串行步骤流程
- 把完全不同任务硬塞进同一块区域

### 什么时候优先用 `Steps`

`Steps` 适合：

- 用户必须按阶段完成任务
- 阶段边界清晰
- 每一步的输入和产出都能解释清楚

`Steps` 不适合：

- 实际上只是长表单
- 用户经常跳着做、回跳做的任务

## 表单与输入质量

### 占位文本（Placeholder）

占位文本应描述**可输入什么**或**引导用户下一步行为**，而非机械指令。

| 场景     | ✗ 不推荐       | ✓ 推荐                         |
| -------- | -------------- | ------------------------------ |
| 搜索框   | `请输入关键词` | `搜索课程、作业或学生…`        |
| 邮箱字段 | `请输入邮箱`   | `name@school.edu`              |
| 备注字段 | `请输入`       | `为什么拒绝这份作业？（可选）` |

规则：

- 给予具体信息，帮助用户理解可填内容的范围和格式
- 可选字段在 placeholder 末尾标注 `（可选）`
- 搜索框 placeholder 用省略号 `…` 暗示可输入更多

### 结构化空状态

空状态不只是"暂无数据"——它是引导用户下一步行为的节点。

空状态必须包含四个元素：

1. **图标**：`@ant-design/icons`，尺寸 32px，颜色 `--ant-color-text-quaternary`（最弱灰度）
2. **标题**：用 `Typography.Text strong`，不用 `Title`——空状态不是页面主层级
3. **说明**：用 `Typography.Text type="secondary"`，限 `maxWidth: 360` 防止过宽，解释为什么为空及用户可以做什么
4. **操作入口**：至少一个主操作按钮——不能只给用户看一个"无数据"然后让他自己猜下一步

**禁止**：只放一行灰字 `暂无数据`。

## 反馈建议

### 轻反馈

适合：

- 成功保存
- 轻量状态更新
- 可撤销的低风险动作

优先考虑：

- `antd message`
- 按钮 loading
- 局部状态变化

### `message` 与 `notification` 的选择

- **`message`**：适合操作直接反馈（成功/失败提示），全局轻提示，自动消失，无需用户交互，不含额外动作
- **`notification`**：适合需要标题 + 说明的通知、含操作按钮的通知、系统级推送或后台任务异步完成通知

通常：单次 CRUD 操作反馈用 `message`；需要阅读内容或包含跟进动作的通知用 `notification`。

禁止：用 `notification` 代替 `message` 做每次操作反馈（过重）；用 `message` 承载需要用户处理的复杂消息（过短）。

当前项目补充：

- refresh 成功 / 失败这类短句状态反馈，统一按 `message` 处理
- 若刷新后马上发生页面跳转，提示应由 layout 内稳定宿主承接，不挂在即将卸载的页面局部宿主上
- refresh 失败若已决定跳登录，不再额外把同一原因同时写成登录表单错误和 toast，避免双提示

### 持续反馈

适合：

- 需要解释原因的警告
- 页面内持续存在的问题
- 需要用户稍后处理的状态

优先考虑：

- `Alert`
- 卡片内说明区
- 空状态说明区

已落地参考：

- `src/features/auth/ui/login-form.tsx` 登录失败用 `<Alert type="error" showIcon title={errorMessage} />`
- `src/features/public-auth/ui/reset-password-intent-panel.tsx` 重置链接校验失败用 `<Alert type="error">` + 描述 + 操作按钮
- `src/app/layout/entry-sidecar.tsx` Sidecar 降级状态用 `<Alert type="info|warning">` 解释当前状态
- `src/app/layout/app-layout.tsx` labs 路由警告用语义化 wrapper（`border-warning-border bg-warning-bg`）+ `Typography.Paragraph`

### 加载反馈

适合：

- 页面骨架加载
- 卡片局部加载
- 提交中状态

优先考虑：

- `Skeleton`
- `<Button loading>`
- 必要时 `Spin`

不要混用多种加载语言，让同一区域同时出现骨架和转圈。

已落地参考：

- `src/pages/home/index.tsx` 首页模块加载用 `<Skeleton active>` 占位（3 张卡片骨架）
- `src/features/api-health-status/ui/api-health-status-panel.tsx` 健康检测加载用 `<Skeleton>`
- `src/app/router/index.tsx` 会话恢复中用 `<Spin size="large">` 全屏居中
- 登录/重置密码表单提交用 `<Button loading>` 的 `submitting` 状态

### AI 生成反馈

AI 场景不要和普通 HTTP loading 混为一谈。

项目中 AI 协作的可用性由 `CollaborationAvailability` 类型管理（`src/app/providers/collaboration-session.ts`），拆为四种状态：

| 状态          | 含义                 | UI 表现                                                         |
| ------------- | -------------------- | --------------------------------------------------------------- |
| `available`   | 正常可用             | Sidecar 正常交互，显示引导文案                                  |
| `degraded`    | 不稳定，部分能力受限 | `<Alert type="warning">` 提示降级，Sender 仍可输入              |
| `readonly`    | 只读，不接受新输入   | `<Alert type="info">` 提示只读，Sender 禁用，历史消息保留       |
| `unavailable` | 完全不可用           | `<Alert type="info">` 提示暂未连接，Sender 仍可输入基础导航意图 |

对于 AI 内容生成过程，更适合拆成三种状态：

- `Thinking`：模型已接收任务，但还没有可展示内容
- `Streaming`：内容正在持续生成
- `Complete / Interrupted`：生成完成或被用户中断

建议：

- `Thinking` 优先用轻量占位、状态文案或思考态，不必沿用整块骨架屏
- `Streaming` 优先保留已生成内容，让用户看到内容逐步出现，而不是一直转圈等待
- AI 生成过程应尽量允许中断、重试、重新生成
- 长生成过程中，状态反馈应稳定、克制，不要频繁闪烁或跳动
- 不要用 `Spin` 把整块 AI 区域长时间盖住，让用户看不到已生成部分

## 确认边界建议

以下情况通常应有显式确认：

- 删除、作废、覆盖、清空
- 会影响多条数据的批量操作
- 不可逆操作
- AI 触发的实际业务写入

以下情况通常不必强行弹确认：

- 可立即撤销的轻操作
- 只是查看或展开
- 低风险偏好设置

不要把"每次都先弹确认"当成默认安全策略。更好的策略通常是：

- 低风险操作减少阻断
- 高风险操作提高确认强度

### 确认强度的选择：`Popconfirm` vs `Modal`

- **`Popconfirm`**：适合单条行内确认（如表格单行删除），后果直观，用户无需阅读额外说明
- **`Modal.confirm` / 完整 `Modal`**：适合需要解释后果、影响多条数据、或用户必须阅读风险提示才能决策的场景
- 不要把 `Popconfirm` 用于批量操作或高风险不可逆场景
- 不要把 `Modal` 用于每一个单行删除——确认感过重会降低操作效率

## 反向速查

### 如果你下意识想到某个组件，先检查是不是误用

- 想用页面：
  先问它是否真的需要回访、深链接、复杂流程或长阅读。若只是临时独立任务，更可能是 `Modal` 或 `Drawer`。
- 想用 `Modal`（用于确认）：
  先问这是不是单条、后果直观的行内操作。若是，更可能是 `Popconfirm`。再问用户是否需要持续对照背景内容。若需要，更可能是 `Drawer` 或 inline。
- 想用 `Drawer`：
  先问任务是否其实只是一次短确认。若是，更可能是 `Modal`。再检查 EntrySidecar 是否已打开——若是，需要考虑两个面板的并发策略。
- 想用 inline：
  先问它是否已经变成多字段、有关联校验的编辑。若是，不要继续 inline，改用 `Drawer` 或页面。
- 想用 `Popover`：
  先问里面是不是已经塞进长内容、关键确认或复杂焦点流。若是，不要再坚持轻容器。
- 想用 `Tooltip`：
  先问它是不是已经承载了主信息或需要认真阅读的说明。若是，改用 `Popover`、inline 或正文区。

## AI 场景补充建议

对于当前项目：

- Sidecar 更适合问答、解释、推荐、预审和上下文补充
- 主业务唯一入口不应塞进 Sidecar
- AI 给建议时，可先在 Sidecar 或 inline 区块展示，再由用户确认进入正式写入

### Sidecar 何时升级到第三工作区

结合当前项目的 layout，AI 结果物不必永远停留在 Sidecar 中。

更适合继续留在 Sidecar 的内容：

- 短问答
- 短摘要
- 单段解释
- 少量候选操作（如 entry card 导航）

更适合升级到第三工作区（Artifacts / Canvas）的信号：

- 代码片段已经接近或超过 Sidecar 的舒适阅读范围
- 表格、结构化清单或对比内容需要明显更宽的横向空间
- 内容需要连续审阅、复制、对照、批注或二次操作
- Sidecar 已经开始出现频繁横向滚动或超长连续滚动

可作为建议信号，而不是硬规则：

- 代码超过约 30 行
- 表格超过约 5 列
- 结果物已经从"回复"变成"文档 / 草稿 / 画布"

已落地参考：`src/labs/demo/page.tsx` 和 `src/shared/third-workspace-demo/canvas.tsx` 演示了从主区/Sidecar 到第三工作区的跳层流程。

简单说：

- Sidecar 负责协作入口与上下文
- 第三工作区负责庞大结果物的展开审阅

### 多面板并发时如何判断

当前项目同时存在 `main`、Sidecar 和潜在的第三工作区，因此不要只按单容器思路设计。

建议优先级：

- 阻断性确认与高风险模态，优先级最高
- 主业务编辑面板，其次
- Sidecar 的持续协作上下文，再其次

建议做法：

- 不要让两个"边缘滑出的大面板"长期并排竞争注意力
- 若 Sidecar 已打开，而主业务又需要打开一个业务 `Drawer`，优先让 Sidecar 收窄、退到上下文态，或暂时关闭
- 若出现阻断性 `Modal`，应优先冻结背景交互，并让焦点和 `Esc` 优先服务该模态
- 第三工作区打开后，Sidecar 更适合继续承担上下文控制，而不是继续承载大结果物正文

目标不是永远禁止并发，而是避免用户同时处理两个同等级工作面。

### AI 错误与降级恢复

AI 交互不只要定义"成功时怎么显示"，也要定义"失败后怎么继续"。

当前项目的 `CollaborationAvailability` 已包含降级路径：

| 异常类型     | 当前 UI 策略                                             |
| ------------ | -------------------------------------------------------- |
| 入口暂未连接 | `unavailable` → Alert 提示，Sender 仍可输入基础导航意图  |
| 入口降级     | `degraded` → Alert 警告，优先回退到本地入口              |
| 只读模式     | `readonly` → Alert 提示，Sender 禁用，历史消息保留为只读 |
| 正常可用     | `available` → 正常交互                                   |

通用建议：

- 已有历史内容优先保留为只读，不在失败时整块清空
- 错误反馈尽量局部化，优先放在当前 AI 交互区，而不是用全局报错淹没主界面
- 为用户提供明确下一步，如重试、重新生成、复制已有内容、回到手动流程
- 对安全拦截或权限限制，应说明"为什么不能继续"，而不只显示失败
- 当 AI 不可用时，主业务流程应继续可用；Sidecar 可退化为说明态、只读态或禁用态

更简单地说：

- 解释和建议，可以轻一些
- 真正写业务数据，必须重一些
- AI 失败时，优先保住上下文，再提供恢复路径

## 一句话判断

拿不准时，先用这个顺序判断：

1. 这是独立任务，且需要复用、回访、深链接或复杂流程吗？是，才优先页面；只是临时独立任务，优先考虑 `Modal` 或 `Drawer`。
2. 需要保留背景上下文吗？是，就优先 `Drawer` 或 inline。
3. 只是短确认吗？是，就优先 `Modal`。
4. 只是补一句说明吗？是，就优先 `Tooltip` 或 `Popover`。

如果一个方案让用户"看不清背景、记不住上下文、又要做很长的事"，那它大概率选错了容器。
