# UI Icons

UI Icon 指产品内部用于导航、操作、状态和提示的功能图标。

## 来源

- 统一使用 `@ant-design/icons`
- 不引入第二套常驻功能图标库
- Logo 与浏览器 / app 图标不进入业务交互控件
- 若涉及本地图标资产目录、运行时交付或 SVG 清理，继续看 [asset-engineering.md](./asset-engineering.md)

## 尺寸

默认只保留三档：

- 文本同行小图标：`1em`
- 按钮 `icon`：跟随 antd Button 默认尺寸
- icon-only 按钮：仍使用 antd Button，不裸放图标

约束：

- 不单独发明一套 icon 尺寸系统
- 需要更强视觉体量时，优先升容器层级，不优先无节制放大 icon
- 图标与文字同行时，统一用结构对齐方式处理，如 `flex items-center`；禁止用 `mt-0.5`、`translate-y` 之类魔数微调图标基线

## 颜色

- 默认继承当前语义颜色，不手写魔法值
- AI 区可用 `--color-ai-accent` 表达 AI 专属语义，但不扩散成通用功能色
- 小号功能图标必须保证可见性，不为了品牌感强行降对比

## 交互状态

- 图标的 hover / active 反馈应保持克制，不用弹跳、持续旋转、无意义缩放这类抢注意力的动效
- 默认优先使用透明度、轻微颜色反差或容器背景变化提供反馈
- 若图标已经处在 Button、Dropdown、Tabs 等 antd 容器里，优先复用容器本身的交互反馈，不再给图标叠第二套表演性动画
- 不打扰任务的动效，通常比“存在感很强”的动效更高级

## 密度与装饰预算

以下为建议，不是硬规则：

- 若标题、状态、按钮都已表达清楚，不再额外补装饰 icon
- 同一行内若已经存在 leading icon，就谨慎再加状态 icon、跳转 icon、装饰 icon，避免语义冲突和视觉噪音
- 一个 surface 内尽量只保留一种主要装饰手法；如果已经有 icon，就少再叠角标、描边、底纹、徽章

## 可访问性

- 只有图标、没有可见文本的按钮，必须提供 `aria-label`
- 含义不直观时，建议补 `title` 或 `Tooltip`
- 纯装饰图标应 `aria-hidden`

## 语义

- 导航、操作、状态、提示四类语义优先复用稳定图标
- 不要把状态 icon 和操作 icon 混用
- 状态反馈优先使用 antd 内建组件语义，再决定是否补额外 icon

## 视觉判断句

- 如果一个标题必须靠 icon 才显得像标题，通常应先回头检查结构和排版
- 当克制地移除了多余图标后，优先依赖字重、文字对比度和留白来建立层次；排版和间距本身就是最稳定的设计语言
