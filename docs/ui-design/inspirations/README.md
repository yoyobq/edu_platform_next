# UI Inspirations

本目录用于存放外部 UI 风格参考与项目内吸收判断。

它不是项目设计规范的主入口，也不是页面模板目录。
这里的内容只负责回答两个问题：

1. 外部有哪些风格值得参考
2. 这些风格在本项目中应该如何吸收、如何止步

## 目录角色

本目录属于 `docs/ui-design/` 的参考层，不属于规范主干。

它的作用是：

- 为设计判断提供高标准、现代化的外部参照物
- 帮助 AI 和开发者理解“像什么”与“不像什么”
- 记录某些风格为何适合本项目、为何不适合本项目
- 为特殊高光页面（如空状态、引导页、发布成功页）提供有限的风格借鉴来源

它不负责：

- 定义全局 token（由 antd theme 决定）
- 定义组件视觉规范
- 定义页面骨架结构
- 取代品牌调性判断
- 直接提供可复制的页面实现方案

## 与主规范的关系

本目录是参考层。当本目录与项目正式规范冲突时，始终以正式规范为准：

1. `ui-stack-rules.md`
2. `layout.md`
3. `ui-design/` 下的正式规则文档
4. `inspirations/` 下的外部参考文档

换言之，外部风格只能被吸收并转化为符合项目基调的产物，不可反向破坏既有秩序。

## 使用原则

### 1. 只能吸收，不可照搬

本目录记录的是“可借鉴的交互模式与视觉质感”，不是“仿站目标”。禁止出现以下做法：

- 直接要求页面“做成某网站那样”
- 整体复制外部品牌配色
- 整体复制外部页面结构
- 为了追求炫技而破坏项目现有层级、高密度内容的易读性与品牌主次

### 2. 先匹配项目，再选择参考

本项目的基准气质是现代、高效的 AI 驱动型平台，其审美特征为：

- **极简克制，结构优先**：靠排版比例、对比度与负空间建立层级，而非大量颜色和边框。
- **专业现代，拒绝陈旧**：摒弃传统后台（如早期 Bootstrap/早期 IBM Carbon）的沉闷感与笨重阴影。
- **高密度下的呼吸感**：在信息密集的列表和表单中，依然保持视觉上的清爽。
- **AI 元素自然融入**：AI 作为 Sidecar 或内嵌工具，视觉上要有特殊质感（如专属强调色或微妙光晕），但不喧宾夺主。
- **细节见真章**：依靠完美的 8px/4px 间距节奏、丝滑的微动效（如 hover 状态、骨架屏展开）提升质感。

### 3. 参考应拆解为局部能力

吸收外部风格时，应尽量拆解为以下层面：

- 页面骨架与信息密度
- 留白（Spacing）、边距与圆角（Radius）节奏
- 卡片层级（Elevation）与光影（Shadows/Glows）
- 焦点状态（Focus Rings）与交互反馈（Micro-interactions）
- 强调色（Accent Color）使用方式
- AI 子域与主产品区的视觉区隔

### 4. 稀有高光页允许有限跳脱

个别页面（如：欢迎页、重大功能发布、数据大盘空状态）允许在项目总体系内“跳一下”，但必须满足：

- 只用于少量关键节点
- 只有一个主视觉记忆点
- 依靠极佳的排版、插画、毛玻璃（Glassmorphism）或微妙的光晕渐变制造张力
- 绝不破坏全站的主题变量（Token）和语义层级

## 当前参考判断

以下代表了当前现代 Web 平台设计的最高标准。基于本项目气质，我们做出以下吸收判断：

### 一级参考（核心骨架、交互与系统秩序）

#### Vercel (Geist Design)

Vercel 代表了现代 SaaS 平台设计的巅峰，极度克制、优雅且专业。

- **适合吸收**：
  - 极致的排版比例与高对比度文本层级。
  - 克制的无边框/极细边框使用（如 1px subtle border）。
  - 完美的暗黑模式（Dark Mode）映射，以及极佳的光影处理（微妙的 Glow 效果替代生硬阴影）。
  - 组件 hover、focus 状态的丝滑反馈。
- **不宜照搬**：
  - 过于极客的纯黑白高反差（需融入我们自身的品牌色温度）。
- **本项目中的定位**：全站默认骨架、卡片、表单、按钮质感、暗黑模式的基础标杆。

#### Linear

Linear 是高密度数据、键盘优先操作的现代效率工具天花板。

- **适合吸收**：
  - 高密度信息下，利用微小明度差和边界建立的层级秩序。
  - 面向专业用户的焦点状态（Focus states）和键盘导航视觉。
  - 精确、冷峻的控制感与“工具感”。
- **不宜照搬**：
  - 过强的冷色调（让师生或非纯研发场景显得过于疏离）。
- **本项目中的定位**：高密度后台区域、数据列表、表格、命令面板（Command Palette）的克制感与交互反馈参考。

#### Raycast

Raycast 虽然是桌面端，但其 AI 助手与原生工具的融合堪称完美。

- **适合吸收**：
  - 极度紧凑的操作面板与快捷键提示设计。
  - AI 生成内容（Streaming 态）时的打字机排版与加载骨架屏。
  - 列表选择时，丝滑的选中态位移（Magic Motion）。
- **不宜照搬**：
  - 纯 macOS 桌面端组件风格生搬硬套到 Web 端。
- **本项目中的定位**：AI 交互区、快捷操作弹窗、下拉菜单的高级感与动效参考。

### 二级参考（局部场景与特殊气质）

#### Claude

- **适合吸收**：
  - 柔和、陪伴式、阅读体验极佳的 AI 对话容器气质。
  - AI 专属强调色（如克制的琥珀/暖橘色调）与主品牌的优雅分层。
  - 长文本生成的排版舒适度。
- **不宜照搬**：
  - 将整个后台系统做成聊天界面的散漫感。
- **本项目中的定位**：仅用于 AI Sidecar（antdX 承载区）和 AI 助手对话界面的质感参考。

#### Notion

- **适合吸收**：
  - 沉浸式阅读区的留白、暖中性色背景（不那么刺眼的白/黑）。
  - 卡片的轻量感与无压力的输入体验。
- **不宜照搬**：
  - 过于“空白画布化”，导致后台配置表单失去结构约束。
- **本项目中的定位**：富文本编辑器区域、文档详情页、以及需要降低用户心理负担的表单区。

#### Stripe

- **适合吸收**：
  - 关键节点（发布成功、对外入口）的极致高级感。
  - 柔和的高质量渐变与微动效（如精致的 Checkmark 动画）。
- **不宜照搬**：
  - 将复杂的动画与渐变泛滥到日常高频工作台。
- **本项目中的定位**：仅用于极少数入口页、数据概览首屏、付费/成功反馈页的高光点缀。

## 明确不作为主方向的参考

以下风格即使在特定领域很成功，也**绝对不适合**作为本项目的主方向：

- **传统企业级后台（如早期的 IBM Carbon、Element UI）**：过于笨重、生硬的阴影和边框，不够现代。
- **过度 Playful 的风格（如 Duolingo、Figma 营销页）**：过大圆角、过饱和色彩，破坏专业度。
- **拟物化/过度新拟态（Neumorphism）**：浪费屏幕空间，对比度极差，不适合高频效率工具。
- **以情绪氛围替代信息组织的风格**：在我们需要用户快速找到配置项时，展示毫无意义的全屏渐变。

## 建议的目录组织

本目录建议按“单个参考对象一份文档”组织，例如：

- `vercel-geist.md`
- `linear.md`
- `raycast.md`
- `claude.md`
- `notion.md`
- `stripe.md`

每份文档建议包含：

1. 风格关键词
2. 核心可吸收点（含视觉与交互）
3. 不可照搬点
4. 在本项目中的适用范围（精确到组件层或页面层）
5. 推荐使用场景
6. 禁止误用场景

## 给 AI 的使用要求

读取本目录时，必须遵守以下约束：

- 本目录是参考层，**不是最终规范**。
- **绝对禁止**根据外部风格覆盖现有 Tailwind 配置、Ant Design Token 或 Brand 规则。
- 只能提炼外部风格中的**局部方法**（如：参考 Vercel 的阴影处理，参考 Linear 的焦点态）。
- 页面默认实现仍以项目自身的设计规范（`layout.md`、`ui-stack-rules.md`）为准。
- 若未被明确允许，不得主动引入大面积渐变、多色品牌语法或复杂的动效。
- AI 子域（antdX）可以适度独立呈现 AI 质感，但必须使用项目中定义好的 AI 主题变量，不得与主产品视觉体系割裂。

## 一句话原则

本目录回答“我们可以向谁学习最顶尖的交互与视觉”，但项目的正式规范与代码基座决定了“我们最终落地成什么样”。
