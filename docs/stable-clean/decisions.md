# Stable Clean Decisions

本文件记录当前仓库中，与 `stable` 第二维治理相关的具体判定先例。

## 目标

- 沉淀真实边界案例
- 避免同类问题反复重判
- 让后续的人和 AI 可以直接参考仓库内既有结论

## 记录边界

- 禁止 AI 自行新增或虚构决策条目
- 本文件只记录人类开发者确认，或经过人工核实的真实重构结果
- 若当前问题尚未经过人工确认，不得以“先例”形式写入本文件

## 方法说明

本主题采用“先列清单，再记录具体决策”的方式推进。

含义：

- 清单先提供默认判断框架，见 [checklist.md](./checklist.md)
- 只有当某个问题在当前仓库中真实出现，且判定结果值得保留时，才写入本文件
- 本文件不是开放问题池，也不是理论汇编
- 本文件中的先例可反向推动清单与主规则修订

## 记录要求

每条记录应尽量包含：

- 场景是什么
- 最终判定是什么
- 为什么这样判
- 如有必要，引用相关文件或规则文档

## 记录模板

```md
## <Decision Title>

### 场景

- <当前遇到的边界问题>

### 判定

- <最终落位或处理方式>

### 依据

- <按哪条现有规则判断>
- <为什么不采用其他落位>

### 后续动作

- <是否需要补规则、补清单或调整代码>
```

## 当前状态

## 001: 首页项目列表重构为独立 Feature

### 场景

- `pages/home` 原本直接承载项目状态面板。
- 面板内部同时包含演示数据来源、领域规则和 UI 渲染。
- `entities/project` 下的原始文件同时包含类型、规则和 mock 数据。

### 判定

- 将项目列表面板判定为稳定业务切片。
- 引入第二维。
- 将领域定义保留在 `entities/project/domain`。
- 将演示数据来源、用例和 UI 入口收口到 `features/project-catalog`。
- `pages/home` 只保留页面装配职责。

### 依据

- 该切片已具备明确领域模型：`Project`
- 该切片已具备明确领域规则：`isProjectLive`
- 该切片已具备明确用例：`getVisibleProjects`
- 按现规则，`pages` 只负责组合，不承载具体业务逻辑。
- 演示数据来源不属于 `entity` 的 domain，放入 feature 的 `infrastructure`。

### 后续动作

- 本次重构已完成。
- 该切片可作为后续 `features` 引入第二维的参考模板。
- 后续若出现更复杂的数据源切换，可继续在该 feature 内扩展 `application / infrastructure`。

## 002: 未形成稳定协作用例时，不硬拆协作 Feature

### 场景

- 在评估协作相关治理边界时，审视了 `app/layout`、`app/providers` 和第三工作区相关实现。
- 当前代码中已经存在协作相关实现，但仍以壳层协调、UI 容器和 demo 逻辑为主。
- 尚未出现可命名、可长期维护、去掉 UI 外壳后仍成立的稳定协作用例。

### 判定

- 当前不执行协作 feature 抽离。
- 未形成稳定协作用例时，不把协作相关实现强行提升为 `features/<slice>/`。
- 未达到第二维阈值时，不为了推进治理进度而硬拆新 feature。

### 依据

- 第二维只应用于达到阈值的稳定切片。
- 当前协作相关实现仍以 layout 壳层职责和 demo 形态为主。
- 若强行抽离，会产生名义上的 feature，而不是稳定的业务切片。

### 后续动作

- 等出现稳定协作用例后，再评估是否抽为 `features/<slice>/`。
- `labs -> stable` 迁移时，先规划 `feature` 边界，再决定是否进入第二维。
