# Workbench Entry Rules

本文件是当前 `workbench entry` 主题的直接规则文档。

它只回答以下问题：

- 登录后默认入口与 public / workbench 边界应如何稳定
- 首页工作台与全局壳层、正式页面如何分边界
- 首页模块与首页动作应满足哪些最小约束
- 哪些 demo 语义必须继续被限制在受控实验域内

它不回答：

- 具体登录实现
- 首页模块清单
- 当前有哪些具体业务模块已经接入
- 详细排期与实施顺序

这些内容应继续留在单页 [workbench-entry-plan.md](../plans/workbench-entry-plan.md) 中。

## 适用范围

- 本文件只约束 `workbench entry` 主题
- 它不构成仓库内所有模块统一适用的 plan 拆分方法
- 若与 `layer`、`dependency` 或 `layout` 正式规则发生冲突，按 [rule-precedence.md](./rule-precedence.md) 裁决

## 与现有规则的关系

- `layout.md` 已经定义 `app shell`、`main`、`Sidecar`、第三工作区的正式边界
- 本文件不改写这些全局壳层规则，只补充“登录后默认入口 / 首页工作台”主题语义
- `pages/home` 的具体实现仍需继续服从 [layer-model.md](./layer-model.md) 与 [dependency-rules.md](./dependency-rules.md)

## 核心规则

### 1. 登录后的 `/` 默认进入工作台，而不是欢迎页

- 登录后的 `/` 默认进入用户工作台
- 该工作台用于承接高频入口、关注能力、最近上下文与少量状态概览
- 首页不追求替代所有正式页面
- 首页也不要求把每个页面都缩成一个窗口级缩略态

### 2. 未登录展示页不属于工作台信息架构

- 未登录访问 `/` 时，可由入口层或路由守卫决定跳转登录页，或进入独立的未登录展示页
- 未登录展示页不属于登录后工作台信息架构
- 未登录展示页不默认复用完整工作台壳层
- `/login`、邀请、邮箱验证、重置密码、magic link 等公共入口继续停留在 `public entry shell`
- 登录后不得继续停留在 `/login`
- 普通受保护页回跳继续使用受控 `redirect`
- verification intent 不复用普通 `redirect` 语义

### 3. 全局壳层与首页工作台内容必须分开

- 顶栏、开始按钮、Sidecar、未来 Omni 等跨页面持续存在的能力继续属于 `layout`
- 首页工作台内容继续属于 `pages/home`
- 只有确实具备跨页面持续价值的状态或入口，才允许上升为全局壳层能力
- 首页若需要打开 Sidecar、切入第三工作区或触发协作动作，必须通过受控 action / intent 接入
- 页面不得直接依赖布局私有状态实现这些动作

### 4. 首页模块是摘要与入口，不是正式页面缩略图

- 首页模块更像工作单元摘要
- 它们可以展示状态、最近项、快捷动作，以及进入正式页面的入口
- 不要求每个正式页面都先提供窗口级组件、桌面缩略版或首页嵌入版
- 首页模块默认不承载完整流程
- 复杂筛选、长列表、深度编辑和完整排障继续留在正式页面

### 5. 首页编排受用户态影响，但不得混写不同语义

- 首页默认出现哪些模块，应由登录后的用户态影响
- `权限`：决定某项能力是否可见、是否可进入
- `默认编排`：基于身份、角色或产品策略决定登录后的初始首页结构
- `个人偏好`：由用户自行调整顺序、关注模块或常用入口
- 这三者相关，但不能在接口和实现上混成同一层语义
- 若无用户自定义偏好，首页先降级为基于角色的默认工作台模板
- 若角色模板也尚未建立，仍应提供一个最小默认工作台，避免首页退化为空白页

### 6. 首页模块必须满足统一的最小 contract

- 首页模块必须稳定分为 `visibility / state / entry` 三层
- `visible: false` 的模块不再产出 `state / entry`
- `state` 最少稳定为 `ready / empty / error`
- `entry.primaryAction` 必填，回答“用户从这里最应该先做什么”
- `navigate` 只进入站内受控目标
- `trigger` 只触发受控站内动作
- 首页模块不得额外再维护一套平行的 destination 真相来源
- 默认编排、排序、折叠与个人偏好，不属于单模块 contract
- `shared/home-modules` 只保留 contract type、type guard 与轻量 constructors 这类共享出口
- 首页模板解析、页面级 view model 与具体模块构建逻辑继续留在对应 feature 或页面侧
- 模块的具体 view model 可继续由对应 feature 拥有，但消费面必须保持统一

### 7. 首页模块必须有准入理由

- 首页模块不是默认展示位
- 不应因为“某个页面存在”就自动进入首页
- 至少满足以下条件之一，才考虑进入首页：高频使用、需要持续关注、能提供明确下一步动作、能承接最近上下文、能明显减少一次跳转成本
- 不满足这些条件的能力，优先留在正式页面，不进入首页工作台

### 8. 首页模块默认只作为 UI Adapter

- 工作台上的每个模块默认只作为 UI Adapter，不直接承载复杂业务逻辑
- 模块需要的数据和摘要，应优先通过对应 feature 暴露的 use case 获取
- 若某个首页模块开始承载复杂查询、流程编排或业务判断，应下沉为独立 feature，再由首页消费其公开入口

### 9. 最小默认工作台必须始终成立

- 在没有个人偏好、没有可用角色模板时，首页仍必须给出最小默认工作台
- 最小默认工作台至少回答三件事：
  - 当前状态如何
  - 下一步能做什么
  - 最近上下文当前是否为空
- demo 内容、实验入口、demo query 和实验桥接动作，不得用于填补正式首页兜底

### 10. Sidecar 与页面联动必须保持受控边界

- Sidecar 继续属于全局壳层，不直接改写页面私有状态
- 页面可以向全局协作层暴露受控的 context / state snapshot，供 Sidecar 读取、总结和辅助导航
- 若 Sidecar 需要影响某个页面的表单、筛选、局部流程或其他可写状态，必须通过该页面预先声明的受控 action 进行
- 这套联动默认从“可读上下文”开始，不默认要求每个页面一开始就提供完整动作集合
- 只有确实需要联动的页面，再逐步补充受控 actions 和更深层的协作能力

### 11. demo 语义不得继续污染 stable 路径

- 第三工作区 demo trigger 不得由正式首页直接承接
- demo bridge 继续限制在 `/labs/demo` 等受控实验域
- `workspaceDemo` 只保留 demo 语义，不升级为正式第三工作区 URL 契约
- 离开 demo 实验域时，不应继续把 `workspaceDemo` 传播到首页、正式导航或稳定入口卡片
- 命中 demo bridge 的文案必须显式说明这是 `demo / 验证 / 过渡` 能力
- 第三工作区挂载位本身可以继续作为壳层骨架存在
- 但 demo artifact、demo canvas、demo trigger 与 demo query 都不能自动毕业为正式能力

## 当前不应误做的事

- 不把首页做成传统门户或纯欢迎页
- 不把每个页面强行改造成桌面窗口
- 不把所有状态都提前提升到 `layout`
- 不在登录体系和真实入口未稳定前，过早拍死首页模块清单
- 不让 demo trigger、demo query 或实验入口反向污染正式首页与稳定导航
