# Layout

本文件记录当前项目的 layout 方向与结构原则。

它不是视觉稿，也不是组件清单，而是关于“页面壳层应该如何组织”的正式约定。

## 目标

本项目的 layout 不应退回传统中后台模板：

- 不默认采用常驻左侧菜单
- 不默认采用占位式 footer
- 不把 AI 能力塞进主内容区

目标是形成一套更适合 AI 友好前端的工作台结构：

- 弱导航
- 强内容
- 强 Sidecar

## 核心原则

### 1. `main` 必须单独成立

主内容区必须能够独立承载大部分传统业务页面。

这意味着：

- 关闭右侧抽屉时，业务仍然可以正常浏览与操作
- 页面主信息不能只存在于 AI 抽屉里
- AI 层是增强层，不是主流程必经层

### 2. AI 更适合作为全局 Sidecar，而不是主内容容器

右侧 Sidecar 可以承载：

- 问答
- 页面上下文总结
- 推荐操作
- 快捷跳转
- AI 草稿或建议

但不应承载：

- 页面唯一入口
- 主业务唯一展示位
- 关闭后就无法完成的基础流程

同时应明确：

- Sidecar 不是所有 AI 生成物的最终承载面
- 当生成结果超过 Sidecar 的有效阅读和操作范围时，layout 应允许临时切换到更宽的展示空间
- 这种更宽的展示空间应视为可折叠、可退出、可回到 `main` 的第三工作区，而不是新的默认主布局

同时还应明确：

- `main` 与 Sidecar 不必共享完全相同的内容密度与排版节奏
- `main` 应优先服务于正式业务信息的稳定阅读与操作清晰度
- Sidecar 应允许采用更适合对话、Markdown、代码块、引用块和候选操作的内容组织方式
- 这种差异应服务于阅读效率与上下文理解，而不是制造割裂的双系统体验

### 3. 导航应尽量轻

当前 layout 更适合：

- 顶部轻量状态条 / command bar
- 主内容画布
- 右侧可展开 Sidecar

不适合默认回到：

- 顶栏 + 常驻左栏 + 面包屑 + footer 的传统后台壳

这类结构在信息架构很复杂时可以成立，但对当前项目来说会过早固化成“旧式中后台”。

### 4. 协作入口可以成为菜单的一部分

AI 不一定要以 `AI` 这个词直接出现。

更合理的方向是把它作为“语义化协作入口”：

- `Assistant`
- `Workspace`
- `Guide`
- `Studio`
- `Copilot`

这个入口可以存在于导航体系里，但其行为不一定是“切换到一个传统页面”。

更推荐的交互是：

- 点击菜单项
- 打开右侧 Sidecar / side panel
- 主内容区保持当前业务上下文

### 5. 顶部轻量控制层也可以承担轻量协作入口

顶部轻量控制层不应只承担状态展示。

当前规则应明确：

- 顶部控制层可以同时承担轻量导航、全局状态和轻量协作入口
- 对于“即用即走”的轻任务，不必强迫用户总是打开完整 Sidecar
- 轻量协作入口更适合承接总结当前页、触发简单动作、发起短指令等低摩擦交互
- 这类入口应作为 Sidecar 的补充，而不是替代主协作层

### 6. `main` 与 Sidecar 应有稳定的生命周期边界

layout 不只是静态切片，也要处理路由切换时的状态延续。

因此规则上应明确：

- `main` 的路由切换不应默认销毁或重置 Sidecar 的完整状态
- Sidecar 的挂载节点应位于路由 `<Outlet />` 之外，保持全局级存在
- 当 `main` 从列表页跳转到详情页时，Sidecar 默认保留历史对话，仅更新当前页面上下文
- 只有在用户显式发起“新对话”或切换到不同业务域时，才允许清空 Sidecar

### 7. AI 不可用时，layout 必须能退化到纯业务模式

本项目虽强调 AI 友好，但不能把 AI 可用性当作业务可用性的前提。

因此规则上应明确：

- 当本地模型不可用、外部模型不可达或当前模型能力不足时，已有业务页面仍必须保持可浏览、可导航、可操作
- AI 不可用时，应用应退化为“无 AI 导航、无 AI 自动化、但保留完整业务能力”的模式
- Sidecar 可以隐藏、禁用或降级为只读提示位，但不应让 `main` 失去基础功能
- 若导航中存在语义化协作入口，应允许其在 AI 不可用时变为禁用态、说明态或直接隐藏，而不破坏主导航闭环
- 已生成的 AI 内容若已存在本地上下文，应允许以只读形式保留查看，而不是在失败时整体丢失
- AI 服务恢复后，应优先恢复既有上下文，而不是强制用户刷新或从空状态重新开始
- layout 应为 AI 不可用状态提供明确反馈，但不应让错误提示主导主界面

## 动态行为原则

### 1. Sidecar 应是弹性的，不应预设为固定宽度抽屉

Sidecar 是布局角色，不只是一个固定宽度的 UI 组件。

因此规则上应满足：

- Sidecar 不应被默认理解为固定 `width` 的死板抽屉
- 打开后，`main` 与 Sidecar 应按任务复杂度和上下文密度弹性分配空间
- 关闭时，`main` 应完整回到全宽布局
- 打开时，`main` 内部内容应具备响应式重排能力，不应被粗暴截断
- 必须为 `main` 设定最小有效宽度阈值；达到阈值后，Sidecar 应停止继续挤压 `main`，转而使用自身滚动、折叠或分页来容纳更多内容

### 2. Sidecar 可以对 `main` 施加软影响，但不能未确认改写业务

项目需要避免“完全隔离导致 Sidecar 退化成聊天框”，但也不能让 AI 直接接管主业务。

因此规则上允许：

- Sidecar 高亮 `main` 中的相关区域
- Sidecar 触发滚动定位、聚焦和上下文映射
- 用户主动把 `main` 中的内容送入 Sidecar 作为上下文

同时规则上禁止：

- AI 未经确认直接改写 `main` 中的业务数据
- 把主业务唯一操作入口藏进 Sidecar
- 让页面关闭 Sidecar 后失去基础可用性

### 3. 协作入口可以动态化，但不强制绑定单一形态

协作入口不应被限制成一个永远静止的按钮。

当前正式规则只约束方向：

- 协作入口可以是名词型入口，也可以是动作型入口
- 若系统能识别当前焦点上下文，应优先展示与上下文匹配的协作动作
- 是否采用动态动词、悬浮入口或其他更激进形态，属于后续设计选择，不在当前正式规则内写死

### 4. 三层治理应体现在布局行为上，而不只是颜色或标识

`stable / labs / sandbox` 不只是发布标签，也应影响 layout 中 Sidecar 对 `main` 的作用边界。

当前规则应明确：

- `stable`：保持最保守的增强边界，优先只读、提示、总结、定位
- `labs`：允许更积极的建议、预演和上下文增强，但仍保持显式确认边界
- `sandbox`：允许最开放的实验性协作形态，但不默认跳过关键确认

这意味着环境差异不应只体现在顶栏胶囊或色彩上，也应体现在交互权限和视觉侵入程度上。

### 5. 深度表达也可以成为治理维度，但不得喧宾夺主

layout 的环境差异不只体现在 X/Y 轴的空间分配，也可以通过 Z 轴层级表达协作边界。

当前规则应明确：

- `stable` 中的协作层应更克制、更贴近主业务平面
- `labs` / `sandbox` 可以允许更明显的悬浮感、跨层提示或临时浮层
- 深度表达应服务于边界清晰、权限感知和协作强弱，而不是单纯制造视觉噱头
- 无论采用何种深度表达，`main` 的主业务地位都不应被压制

### 6. Z 轴层级必须由 layout 统一分段治理

一旦同时存在 `main` 自身弹层、Sidecar 容器、Sidecar 内部浮层，以及 `labs / sandbox` 的实验性跨层提示，`z-index` 就会成为高风险污染源。

当前规则应明确：

- Z 轴层级必须按语义角色统一分段管理，而不能由各组件各自发明数字
- `main` 基础层、`main` 内部弹层、Sidecar 容器、Sidecar 内部浮层、治理层跨层浮层，应有明确且互不重叠的层级区间
- 组件开发不应直接写裸 `z-index` 数字，而应统一引用语义化的层级 token 或变量
- `labs / sandbox` 中允许出现更激进的跨层浮层，但仍必须服从全局分段，不得越级污染正式业务层

### 7. Sidecar 打开与关闭时必须保持焦点流连续

当 `main` 与 Sidecar 同时可交互时，必须显式处理焦点与键盘事件的归属。

当前规则应明确：

- Sidecar 打开时，全局焦点应优先平滑转移至 Sidecar 的主输入区
- `Esc` 默认优先用于关闭 Sidecar，而不是触发 `main` 的低优先级取消行为
- 若 `main` 中存在更高优先级的模态层，应由该模态层先接管 `Esc`
- Sidecar 关闭时，焦点应回退到触发它的入口控件，避免键盘流场断裂

### 8. 动效可以作为人机轮转的状态语言，但必须克制

在 AI 协作界面中，动效不应只承担装饰作用，也可以承担状态沟通作用。

当前规则应明确：

- 动效可以用于表达 AI 正在读取、生成、待命、退出或等待确认等状态
- 动效应优先服务于状态理解、节奏感知和控制权边界，而不是单纯制造热闹感
- 与 AI 相关的状态动效应保持克制、连续、可中断，避免打断 `main` 的业务阅读与操作
- 动效表达不应掩盖真实系统状态，关键确认与失败反馈仍需明确可见的文案或结构提示

## 对当前项目的意义

本项目有明确的三层发布治理：

- `stable`
- `labs`
- `sandbox`

因此 layout 不只是视觉容器，也要服务于暴露控制与环境隔离。

在这个前提下：

- `app/` 应承载 layout 壳层与路由组合
- `pages/` 继续承载正式页面内容
- `labs` / `sandbox` 的暴露逻辑不应被 UI 模板掩盖

## 推荐结构

推荐的整体结构接近：

- 顶部轻量控制层
- 中间 `main` 内容区
- 右侧协作型 Sidecar
- 必要时可临时展开的生成物工作区

可选元素：

- 顶部品牌 / 当前环境 / 当前角色
- 少量一级入口
- 语义化协作入口

不推荐默认元素：

- 常驻左栏
- 常规 footer
- 为了形式完整而加的面包屑

## 当前结论

当前 layout 方向结论如下：

- 采用“主业务区 + 右侧 AI Sidecar”
- `main` 必须可独立运行
- AI 抽屉关闭时，业务不能失效
- 庞大生成物不应被强塞进 Sidecar，必要时可进入临时第三工作区
- 菜单中允许出现语义化协作入口
- 路由切换不应默认重置 Sidecar，Sidecar 应作为全局级协作层存在
- AI 不可用时，layout 必须能退化为纯业务模式
- AI 不可用时，应尽量保留既有只读上下文，并在恢复后延续会话
- Sidecar 允许对 `main` 施加软影响，但不允许未确认直接改写业务
- Sidecar 的弹性扩张必须服从 `main` 的最小可用宽度边界
- Z 轴深度也可以参与表达治理边界，但不得压过主业务平面
- Z 轴层级必须由 layout 统一分段治理，组件不得随意写裸 `z-index`
- Sidecar 打开与关闭时必须保持明确的焦点归属与回退路径
- `stable / labs / sandbox` 的差异应体现在 layout 行为边界上
- 暂不采用传统中后台壳作为默认 layout
